# ポートフォリオ サイト設計書（情報設計 / 回遊性 v1）

対象リポジトリ: `minoru-s/portfolio` / 現公開URL: https://minoru-s.github.io/portfolio/

---

## 0. この文書の位置づけ・前提（最初に必ず読む）

- **目的:** 現在の「1枚縦スクロール（詳細はモーダル）」のサイトを、**"深く知りたい人が潜って回遊できる"多階層サイト**に作り替えるための情報設計（IA）仕様。
- **コンテンツ本文は別文書 `portfolio_update_requirements.md`（v2）で確定済み。** 本書はその本文を「どのページに置き、どう繋ぐか」を定義する。**必ず両方を併せて参照すること。**
- 本書は**ビジュアル（配色・モーション）の再設計ではなく、構造とナビゲーションの設計**。既存の Liquid Glass / Tailwind の見た目は踏襲・移植する。
- 現状スタック: vanilla HTML + Tailwind + GitHub Pages、日英二言語（`portfolioDataJa` / `portfolioDataEn` の2オブジェクト）。
- **読者は2種類。設計は常にこの二層を両立させること:**
  - **A: 採用担当（30秒スキム）** … ホーム/概要が単独で完結し、各詳細ページも冒頭で「何 / なぜ / 結果」が即読める。
  - **B: 深掘りする人（エンジニア・将来の共同研究者・興味を持った採用担当）** … リンクを辿って rabbit hole に潜れる。

---

## 1. 設計の核（3原則）

1. **作品ごとに固有URLの詳細ページを持つ**（モーダルは廃止）。共有・SEO・ESでの個別リンクのため。
2. **どのページにも"次に行ける出口"を必ず置く**（関連リンク＋バックリンク＋本文中インラインリンク＋タグ）。**袋小路を作らない。**
3. **二層構造を崩さない**（浅い層＝シャープに保つ、深い層＝枝として増やす）。

## 3. ページ別仕様

### 3.1 Home `/`
- 役割: ロビー。**スキムで完結すること（A層）。**
- 縦構成: Hero（名前 ＋ 一言の価値提案「ハード×ソフトを作るR&D志望」＋ CTA） → About要約（3〜4行） → Skills（現行を流用） → **Featured Work（厳選4〜6カード → 各詳細ページへ遷移）** → Highlights/Achievements（受賞・メディア・主要数字） → 最新の Writing 3件 → Contact。
- **重要: Featured カードはモーダルでなく詳細ページへ遷移させる。**
- Hero は演出で中身を埋めない。オープニング演出を入れるならスキップ可能にする。

### 3.2 プロジェクト / 研究 詳細ページ（共通テンプレート）
全詳細ページで構造を統一する（上から。スキム両立のため）:
1. タイトル ＋ 一言サマリ（＝何をしたか）
2. メタ情報: 役割 / 期間 / 技術スタック（chips） / 主要リンク（demo・GitHub 等）
3. ヒーローメディア（画像 / 動画 / 3D）
4. 本文: **結論 → 背景・課題 → やったこと（自分の判断を1つ） → 結果**（西藤さんのES構造に準拠）
5. （該当時）**ライブ埋め込みデモ**（`/lab/` の島を埋め込む）
6. ギャラリー
7. **関連（Related）** ＋ **バックリンク（このページに言及している記事/プロジェクト）**
8. 前へ / 次へ
- **冒頭2〜3行で「何 / なぜ / 結果」が読めること**（B層が潜る前にA層が判断できる）。

### 3.3 一覧ページ `/projects/` `/research/` `/writing/`
- カードグリッド ＋ タグ/種別/年フィルタ ＋（任意）検索。
- `/writing/` は新しい順だが、digital garden 的に「育成中」バッジ等で日付依存を弱める（陳腐化対策）。

### 3.4 `/lab/`
- ツールを説明でなく**操作させる**ハブ。Mocap Plus / YuruTask を埋め込み（または全画面リンク）。**デモデータを同梱**して即触れるようにする。
- 各プロジェクト詳細ページ ↔ `/lab/` の該当ツールを相互リンク。

### 3.5 `/writing/`（ブログ ＝ ビルドログ / ノート）
- 種別: 「作ってみた」 / 技術メモ / 研究解説。
- **初期は卒論付録・既存ESの素材を再利用して3〜4本**（ゼロから書かない）。例: 跳躍パッド30案の試行錯誤（※最終形状は伏せる）、Mocap Plus の作り方、ローカルLLM環境構築。
- MDX で本文中にツール / 可視化の島を埋め込み可能にする。

### 3.6 `/uses/` `/now/`
- `/uses/`: 3Dプリンタ各機種・ソフト・言語など、実際のツールチェーン（中身が濃いので刺さる）。
- `/now/`: 今やっていること（研究テーマの確定状況・就活・制作中のもの）。月1更新目安。

## 4. 回遊（相互リンク）設計 ← 本書で最重要

- **関連リンク**: 各詳細/記事末尾に、手動 `related[]` ＋ 同タグ自動。
- **バックリンク**: 内部リンクをビルド時に集計し、各ページ末に「このページに言及しているもの」を自動表示（Andy Matuschak 式）。
- **インラインリンク**: 本文中で言及対象へ繋ぐ（例: 研究ページ → 使った自作ツール `/projects/mocap-plus/` → 関連記事）。
- **タグ横断** `/tags/<tag>/`: projects + research + writing をまたいで束ねる。例: `#motion-capture` を押すと「月面研究 ＋ Mocap Plus ＋ 解析の技術メモ」が一覧化される。**これが回遊の主動線。**
- **前へ / 次へ** ＋ 永続グローバルナビ ＋ 迷子用インデックス/検索（袋小路対策）。
- タグ taxonomy（初期案・本人が確定）: `robotics` / `haptics` / `motion-capture` / `simulation` / `web` / `tools` / `3d-printing` / `data-analysis` / `video` / `space`。

## 5. 技術スタック（推奨）と移行

- **推奨: Astro。** 理由: content collections（markdown/MDX で projects・research・writing を型付き管理）、MDX で本文中のインタラクティブな島だけ hydrate できる（重い3D/デモを入れてもページは軽量を維持）、静的出力で **GitHub Pages のまま配信**、高速。バックリンクはビルド時生成が可能。
- 軽量代替: Eleventy（digital garden 系テンプレートが豊富）。
- **非推奨（最小工数）**: 現行 vanilla のまま各ページを手書き ＋ 簡易ビルド。ブログ/バックリンク/タグ横断の拡張に弱いので、回遊性を本気で出すなら避ける。

**データモデル（Astro content collections の frontmatter 例）**
各 projects / research / writing エントリに以下を持たせる:
```yaml
titleJa: ""
titleEn: ""
slug: ""
date: 2025-01-01
updated: 2025-01-01
tags: []
summaryJa: ""      # 一言サマリ（＝何）
summaryEn: ""
role: ""           # 役割（projects/research）
period: ""         # 期間
stack: []          # 技術スタック chips
hero: ""           # ヒーロー画像/動画パス
demoUrl: ""        # 任意（匿名対象は空）
repoUrl: ""        # 任意
draft: false
related: []        # 手動の関連slug
```

**二言語対応**
- Astro i18n で `/`（日本語＝既定）と `/en/` に**全ルートをミラー**。本文はロケール別 MDX。
- **全ページで日英パリティ必須**（現行同様）。
- 既存 `portfolioDataJa` / `portfolioDataEn` の中身は、本書 3章のページ単位に展開して content collections へ移送する。

## 6. 引き継ぐ制約（リビルドで絶対に失わないこと）〔重要〕

`portfolio_update_requirements.md`（v2）と同じ制約を、新サイトでも厳守:
- **学生Webメディア**: 媒体名・URL・SNS・**具体ツール名（GPA計算機/自販機マップ等）・出身大学への紐付け**・実サイトのスクショを載せない。`links: []`。
- **跳躍パッド**: 最終形状の詳細（通称名・形状名）を書かない。**過程と結果のみ。**
- **プログラミングスクール**: 匿名（「大手プログラミングスクール」）維持。
- **取り違え厳禁**: シミュレーション（群探査評価）と跳躍パッド（実機の試作）を混同しない。
- **日英パリティ**維持。
- 画像 `webmedia.png` は中身が特定できない抽象画像（スクショ不可）。

## 7. 構築フェーズ（最小で効く順）

就活の時間制約を踏まえ段階的に。**各フェーズ単体で公開可能**にすること。
- **Phase 1（骨格・最優先）**: Astro 移行 ＋ Home ＋ projects/research の詳細ページ化（モーダル廃止）＋ 一覧ページ ＋ グローバルナビ。← **これだけで「回遊できる」骨格が完成。**
- **Phase 2（回遊）**: 関連リンク ＋ 本文中インラインリンク ＋ タグ ＋ タグ横断ページ ＋ バックリンク。
- **Phase 3（深さ）**: `/writing/` 立ち上げ（既存素材から3〜4本）＋ `/lab/`（ツール埋め込み）。
- **Phase 4（生きてる感）**: `/uses/` ＋ `/now/` ＋（任意）検索。
